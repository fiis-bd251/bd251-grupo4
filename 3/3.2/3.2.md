# 3.2. Módulo 2

## Casos de uso:

### 1. Identificación del Dolor

| **Realised User Task**                     | **Identificación del dolor – Requerimiento: OPP-001**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Responsable de identificar el dolor o problema del cliente.                                    |
| **Participating Actors**       | - Ejecutivo Comercial<br>- Sistema de Gestión de Oportunidades (para registrar los datos)                           |
| **Flow of Events**             | 1. El Ejecutivo Comercial interactúa con el cliente para entender los principales problemas (dolores) que enfrenta la empresa.<br>2. El Ejecutivo Comercial registra los problemas identificados en el sistema.<br>3. El sistema almacena la información y la relaciona con la oportunidad. |
| **Exceptions**                 | Si no se puede identificar un dolor claro, el proceso de calificación no avanza.                                     |
| **Preconditions**              | El cliente debe estar registrado en el sistema y debe haber interacción con un representante comercial.             |
| **Postcondition**              | El dolor del cliente ha sido identificado y registrado, permitiendo avanzar a la validación del stakeholder.         |
| **Includes Use Case**          | Validación del stakeholder: El dolor identificado debe estar relacionado con la persona correcta en la organización. |
| **Used Services**              | Servicios de análisis y registro de problemas del cliente.                                                          |
| **Non-functional requirements**| Usabilidad: La interfaz debe ser fácil de usar para registrar los problemas.<br>Seguridad: Los datos deben ser confidenciales. |



### 2. Validación del Stakeholder

| **Realised User Task**                     | **Validación del stakeholder – Requerimiento: OPP-002**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Responsable de interactuar con el cliente y determinar si está tratando con la persona correcta. |
| **Participating Actors**       | - Ejecutivo Comercial<br>- Sistema de Gestión de Oportunidades (utiliza los datos registrados para realizar validaciones) |
| **Flow of Events**             | 1. El ejecutivo comercial interactúa con el cliente para validar su rol dentro de la organización.<br>2. Si el cliente tiene poder de decisión, se continúa. Si no lo tiene, el ejecutivo solicita acceso a la persona con autoridad. |
| **Exceptions**                 | No hay acceso a la persona con poder de decisión, falta de información.                                             |
| **Preconditions**              | La necesidad del cliente ya ha sido registrada en el sistema.                                                      |
| **Postcondition**              | El cliente o stakeholder con poder de decisión ha sido identificado, permitiendo pasar a la siguiente etapa de calificación. |
| **Includes Use Case**          | Registro de necesidad o dolor, ya que la validación de quién está involucrado es parte de la calificación.          |
| **Used Services**              | Servicio de validación de stakeholders: Permite verificar la autoridad del interlocutor.                             |
| **Non-functional requirements**| Seguridad: Solo los ejecutivos autorizados pueden acceder a información confidencial.<br>Audibilidad: Todas las interacciones quedan registradas para futuras referencias. |



### 3. Información de la Convocatoria

| **Realised User Task**                     | **Información de la convocatoria – Requerimiento: OPP-003**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Gerente Comercial: Responsable de registrar la información relacionada con la convocatoria.                         |
| **Participating Actors**       | - Gerente Comercial<br>- Sistema de Gestión de Oportunidades (para almacenar los datos de la convocatoria)          |
| **Flow of Events**             | 1. El Gerente Comercial recibe la información relacionada con la convocatoria del cliente.<br>2. El Gerente Comercial registra los detalles en el sistema, incluyendo fecha de publicación, fecha límite, y tipo de convocatoria.<br>3. El sistema almacena la información y la vincula con la oportunidad. |
| **Exceptions**                 | Si falta información o si la convocatoria no está formalmente definida, el proceso de calificación se detiene.      |
| **Preconditions**              | La necesidad del cliente y la validación del stakeholder deben estar completadas.                                    |
| **Postcondition**              | La información de la convocatoria ha sido registrada correctamente y está lista para el análisis técnico.           |
| **Includes Use Case**          | Análisis técnico: La información de la convocatoria debe ser evaluada para determinar la viabilidad.                 |
| **Used Services**              | Servicio de registro y análisis de convocatorias.                                                                   |
| **Non-functional requirements**| Seguridad: La información de la convocatoria debe ser confidencial.<br>Usabilidad: El sistema debe permitir un ingreso fácil de los datos de la convocatoria. |



### 4. Análisis Técnico

| **Realised User Task**                     | **Análisis técnico – Requerimiento: OPP-004**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Equipo Técnico: Responsable de analizar la viabilidad técnica de la oportunidad y determinar si se necesita un partner. |
| **Participating Actors**       | - Equipo Técnico<br>- Sistema de Gestión de Oportunidades (para registrar el análisis técnico)                      |
| **Flow of Events**             | 1. El equipo técnico recibe la solicitud de la oportunidad.<br>2. Se analiza si la solución es viable desde un punto de vista técnico.<br>3. Si se requiere un partner, se registra en el sistema y se evalúa su capacidad técnica. |
| **Exceptions**                 | Si la solución no es viable técnicamente, el proceso de calificación no continúa.                                   |
| **Preconditions**              | La información de la convocatoria debe estar registrada.                                                           |
| **Postcondition**              | El análisis técnico ha sido completado, permitiendo avanzar al siguiente paso (registro de partner o cierre).       |
| **Includes Use Case**          | Registro de partner: Si es necesario, se registra un partner para la implementación de la solución.                |
| **Used Services**              | Servicios de evaluación técnica y registro de partner.                                                              |
| **Non-functional requirements**| Seguridad: La evaluación técnica debe ser confidencial.<br>Audibilidad: El análisis técnico se guarda para referencia futura. |



### 5. Registro de Partner

| **Realised User Task**                     | **Registro de Partner – Requerimiento: OPP-005**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Responsable de registrar un nuevo partner si la solución lo requiere.                          |
| **Participating Actors**       | - Ejecutivo Comercial<br>- Sistema de Gestión de Oportunidades (para registrar el partner)                           |
| **Flow of Events**             | 1. El Ejecutivo Comercial identifica la necesidad de un partner para ejecutar la solución.<br>2. Se registran los detalles del partner en el sistema (nombre, contacto, especialidad, etc.).<br>3. El sistema valida y almacena la información del partner. |
| **Exceptions**                 | Si los datos del partner son incorrectos o incompletos, se solicita corrección antes de registrar la información.    |
| **Preconditions**              | El análisis técnico debe haber confirmado la necesidad de un partner.                                               |
| **Postcondition**              | El partner ha sido registrado y está disponible para la implementación de la solución.                              |
| **Includes Use Case**          | Análisis técnico: El partner se vincula al análisis técnico si es necesario.                                        |
| **Used Services**              | Servicio de registro de partners y validación de sus datos.                                                         |
| **Non-functional requirements**| Seguridad: La información del partner debe ser confidencial.<br>Audibilidad: Toda la información registrada es accesible para futuras referencias. |



### 6. Aprobación Final

| **Realised User Tasko**                     | **Aprobación final – Requerimiento: OPP-006**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Gerente Comercial: Responsable de obtener la aprobación final del cliente.                                          |
| **Participating Actors**       | - Gerente Comercial<br>- Cliente                                                                                     |
| **Flow of Events**             | 1. El Gerente Comercial se comunica con el cliente para obtener su aprobación de la propuesta.<br>2. El cliente revisa y aprueba la propuesta.<br>3. Si la propuesta es aprobada, el proceso avanza; si no, se solicitan ajustes. |
| **Exceptions**                 | Si el cliente no aprueba, se realizan ajustes y se reenvía la propuesta.                                            |
| **Preconditions**              | La propuesta debe haber sido generada y enviada previamente.                                                       |
| **Postcondition**              | La propuesta ha sido aprobada y la oportunidad se marca como cerrada o se pasa a negociación.                       |
| **Includes Use Case**          | Generación de propuesta comercial: Necesaria para la fase de aprobación.                                            |
| **Used Services**              | Servicios de seguimiento de propuestas y comunicación con el cliente.                                              |
| **Non-functional requirements**| Confidencialidad: Los términos de la propuesta deben mantenerse privados hasta la aprobación final.                 |



### 7. Resumen del Proceso

| **Realised User Task**                     | **Resumen del proceso – Requerimiento: OPP-007e**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Responsable de generar un resumen final del proceso de calificación de la oportunidad.         |
| **Participating Actors**       | - Ejecutivo Comercial<br>- Gerente Comercial<br>- Sistema de Gestión de Oportunidades (para almacenar y generar el resumen) |
| **Flow of Events**             | 1. El Ejecutivo Comercial, al finalizar el proceso de calificación, accede a la opción de generar un resumen del proceso.<br>2. El sistema recopila la información de todas las etapas (identificación del dolor, validación del stakeholder, análisis técnico, propuesta, validación de partner, etc.) y la organiza en un formato claro y resumido.<br>3. El Ejecutivo Comercial revisa el resumen y realiza ajustes si es necesario.<br>4. El resumen final se guarda en el sistema y se puede exportar como PDF para su envío al cliente o almacenamiento interno. |
| **Exceptions**                 | Si falta alguna información en las etapas previas del proceso, el resumen no puede ser generado hasta que se complete la información requerida. |
| **Preconditions**              | Todas las etapas del proceso de calificación deben haberse completado previamente (identificación del dolor, validación del stakeholder, análisis técnico, propuesta, validación de partner, etc.). |
| **Postcondition**              | El resumen del proceso ha sido generado y almacenado, permitiendo su consulta futura o envío al cliente.            |
| **Includes Use Case**          | Todos los casos anteriores: El resumen depende de la correcta finalización de las fases de identificación del dolor, validación del stakeholder, análisis técnico, etc. |
| **Used Services**              | Servicios de recopilación de datos del proceso, generación de resumen y exportación de informes.                    |
| **Non-functional requirements**| Usabilidad: El resumen debe ser fácilmente legible y comprensible para cualquier persona involucrada en el proceso.<br>Confidencialidad: Los datos contenidos en el resumen deben ser tratados de acuerdo con las políticas de seguridad. |
