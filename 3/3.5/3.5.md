# 3.5. Módulo 5

## Casos de uso:

### 1. Visualización de Negociaciones


| **Realised User Task**                     | **Visualización de Negociaciones – Requerimiento: NGC-001**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Visualizar y gestionar todas las propuestas, negociaciones abiertas y contratos desde un panel centralizado |
| **Participating Actors**       | - CEO (opcional) |
| **Flow of Events**             | 1. El Ejecutivo Comercial accede a la opción "Dashboard de Propuestas" en el menú principal<br>2. El sistema carga y muestra el panel con todas las propuestas y negociaciones activas<br>3. El sistema muestra listado de propuestas organizadas por etapa (propuestas presentadas, negociaciones activas, contrato por firmar)<br>4. El Ejecutivo puede filtrar por cliente, estado, rango de fechas o valor<br>5. El Ejecutivo selecciona una propuesta específica para ver detalles<br>6. El sistema entra para obtener detalles<br>8. El sistema permite exportar reportes del dashboard en diferentes formatos |
| **Exceptions**                 | - Si hay demasiadas propuestas activas, el sistema pagina los resultados<br>- Si hay problemas de conexión, muestra versión limitada con datos en caché<br>- Si hay propuestas con alertas (próximas a vencer, estancadas), se destacan visualmente |
| **Preconditions**              | El Ejecutivo Comercial está autenticado en el sistema |
| **Postcondition**              | - Se registra la actividad de visualización en logs<br>- Las propuestas con alertas de seguimiento se marcan como revisadas si se accede a ellas |
| **Includes Use Case**          | - Filtrado de propuestas<br>- Generación de reportes<br>- Visualización de timeline |
| **Used Services**              | - Servicio de dashboard analítico<br>- Servicio de búsqueda y filtrado<br>- Servicio de reportes<br>- Servicio de visualización de timeline |
| **Non-functional requirements**| - El dashboard debe cargar en el menor tiempo posible<br>- Los filtros deben aplicarse instantánea y simultaneamente sin recargar la página<br>- Las métricas críticas deben actualizarse automáticamente<br>-Deben verse graficos y cuadros de buena calidad para una correcta visaulización |


### 2. Negociación de Términos

| **Realised User Task**                     | **Negociación de Términos – Requerimiento: NGC-002**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Gestiona los avances en la negociación de términos con el cliente|
| **Participating Actors**       | - CEO (de ser necesario)<br>- Cliente |
| **Flow of Events**             | 1. El Ejecutivo selecciona una propuesta en estado "Pendiente" o "Acordado"<br>2. El Ejecutivo selecciona "Nueva Acción"<br>3. El sistema muestra formulario con los términos actuales<br>4. El Ejecutivo registra cambios en términos (alcance, plazos, entregables)<br>5. El Ejecutivo documenta acuerdos alcanzados y puntos pendientes<br>6. Si hay aspectos técnicos complejos, marca la opción "Solicitar revisión"<br>7. El sistema guarda los cambios y actualiza el estado de la negociación<br>8. Si se solicitó revisión técnica, el sistema notifica al equipo técnico o comercial |
| **Exceptions**                 |  - Si hay términos contradictorios, el sistema advierte al usuario<br>- Si faltan detalles obligatorios, solicita completar |
| **Preconditions**              | - Existe como minimo una propuesta creada<br>- El Ejecutivo tiene permisos sobre la negociación |
| **Postcondition**              | - Registro actualizado del proceso de negociación<br>- Se actualizan los términos de la negociación<br>- Se registra la actividad en el historial<br>- Se notifica a involucrados relevantes |
| **Includes Use Case**          | - Solicitud de revisión técnica (opcional)|
| **Used Services**              | - Servicio de gestión de negociaciones<br>- Servicio de notificaciones<br>- Servicio de comparación de versiones |
| **Non-functional requirements**| - Debe notificar lo mas pronto posible a los involucrados |

#### 4. Negociación Comercial

| **Realised User Task**                     | **Negociación Comercial – Requerimiento: NGC-004**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Crear y/o modifica una accion comercial a un cliente |
| **Participating Actors**       | Área Financiera (opcional)|
| **Flow of Events**             | 1. El ejecutivo comercial selecciona "Nueva Accion" en la seccion de ajustes<br>2. El sistema muestra el formulario de creación de ajuste<br>3. El ejecutivo completa los datos básicos (cliente, descripción, alcance)<br>4. El ejecutivo solicita colaboración del equipo técnico para especificaciones<br>5. El área financiera revisa y completa la sección de precios<br>6. El ejecutivo revisa la propuesta completa<br>7. El ejecutivo envía la accion de ajuste al cliente a través del sistema |
| **Exceptions**                 | - Si faltan datos obligatorios, el sistema muestra un mensaje de error<br>- Si el cliente no existe, se ofrece la opción de crearlo<br>- Si hay inconsistencias en precios, se notifica al área financiera |
| **Preconditions**              | - El cliente debe estar registrado en el sistema<br>- El ejecutivo debe tener permisos para crear acciones<br>- El ejecutivo debe estar correctamente identificado con su rol  |
| **Postcondition**              | - Propuesta almacenada en el sistema y marcada como "Enviada"<br>- Se crea registro de seguimiento asociado a la propuesta |
| **Includes Use Case**          | Consulta de Clientes, Gestión de Presupuesto |
| **Used Services**              | Servicio de Correo Electrónico  |
| **Non-functional requirements**| - El sistema debe permitir trabajar con el tipo de cambio y mostrarlo |

#### 5. Revisión de Propuesta

| **Realised User Task**                     | **Revisión de Propuesta – Requerimiento: NGC-005**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Revisión detallada de aspectos técnicos y comerciales de una propuesta antes de su presentación |
| **Participating Actors**       | Equipo Técnico, Área Financiera |
| **Flow of Events**             | 1. El Ejecutivo Comercial selecciona una propuesta en estado "Borrador"<br>2. El Ejecutivo selecciona "Solicitar revisión técnica y comercial"<br>3. El sistema presenta opciones para seleccionar revisores específicos<br>4. El Ejecutivo asigna revisor técnico y comercial según corresponda<br>5. El sistema notifica a los revisores asignados<br>6. Los revisores acceden a la propuesta y realizan análisis<br>7. Los revisores registran observaciones, sugerencias y correcciones<br>8. Los revisores marcan su revisión como completada<br>9. El sistema notifica al Ejecutivo Comercial cuando todas las revisiones están completas<br>10. El Ejecutivo revisa los comentarios y actualiza la propuesta según sea necesario<br>11. El Ejecutivo marca la propuesta como "Revisada" cuando está conforme |
| **Exceptions**                 | - Si los revisores no completan su revisión en el tiempo asignado, el sistema envía recordatorios<br>- Si hay comentarios contradictorios, el sistema marca la inconsistencia para resolución<br>- Si un revisor rechaza la propuesta por problemas críticos, cambia a estado "Requiere revisión mayor" |
| **Preconditions**              | - Existe una propuesta en estado "Borrador" con información suficiente<br>- Hay revisores disponibles tanto técnicos como comerciales |
| **Postcondition**              | - La propuesta tiene registro de revisiones técnicas y comerciales<br>- La propuesta cambia a estado "Revisada" o "Requiere revisión mayor"<br>- Se genera historial de todas las observaciones realizadas |
| **Includes Use Case**          | - Asignación de revisores<br>- Registro de observaciones |
| **Used Services**              | - Servicio de notificaciones<br>- Servicio de gestión de revisiones<br>- Servicio de gestión documental |
| **Non-functional requirements**| - Las revisiones deben completarse en un plazo máximo de 48 horas<br>- El sistema debe mantener trazabilidad completa de cambios realizados<br>- Debe permitir comparar versiones pre y post revisión<br>- La interfaz debe destacar claramente las secciones con observaciones |

### 6. Agendar Reunión con Cliente

| **Realised User Task**                     | **Agendar reunión con cliente – Requerimiento: NGC-006**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Programar y gestionar reuniones con clientes para presentar y discutir propuestas |
| **Participating Actors**       | Cliente |
| **Flow of Events**             | 1. El Ejecutivo Comercial selecciona una propuesta lista para presentación<br>2. El Ejecutivo elige "Agendar reunión con cliente"<br>3. El sistema muestra calendario con disponibilidad del Ejecutivo<br>4. El Ejecutivo selecciona fecha y hora propuestas, y modalidad (presencial/virtual)<br>5. El Ejecutivo añade detalles como ubicación/enlace y agenda<br>6. El Ejecutivo selecciona participantes internos adicionales (opcional)<br>7. El sistema genera invitación formal para el cliente<br>8. El Ejecutivo personaliza mensaje de invitación y confirma envío<br>9. El sistema envía invitación al cliente con opción de confirmar/rechazar<br>10. Al recibir respuesta del cliente, el sistema actualiza el estado de la reunión<br>11. El sistema envía recordatorios automáticos 24 horas y 1 hora antes<br>12. Tras la reunión, el sistema solicita al Ejecutivo registrar resultados |
| **Exceptions**                 | - Si el cliente rechaza o solicita reprogramación, el sistema facilita proceso de nueva agenda<br>- Si hay conflictos de calendario, el sistema advierte y sugiere horarios alternativos<br>- Si se requieren recursos específicos (sala de reuniones, equipos), el sistema verifica disponibilidad |
| **Preconditions**              | - Existe una propuesta en estado adecuado para presentación<br>- Se cuenta con información de contacto actualizada del cliente |
| **Postcondition**              | - Se crea evento en calendario con todos los detalles<br>- Se envían invitaciones a todos los participantes<br>- Se asocia la reunión al historial de la propuesta y cliente<br>- Se activan recordatorios automáticos |
| **Includes Use Case**          | - Verificación de disponibilidad de recursos<br>- Envío de invitaciones<br>- Programación de recordatorios |
| **Used Services**              | - Servicio de gestión de calendario<br>- Servicio de correo electrónico<br>- Servicio de notificaciones<br>- Servicio de reserva de recursos (opcional) |
| **Non-functional requirements**| - Debe sincronizarse con calendarios externos (Outlook, Google Calendar)<br>- Las invitaciones deben incluir adjuntos relevantes de la propuesta<br>- Debe funcionar en dispositivos móviles para confirmaciones en movimiento<br>- Debe incluir opciones de accesibilidad para reuniones virtuales<br>- El sistema debe enviar recordatorios incluso si el usuario no está conectado |

### 7. Solicitar Elaboración de Contrato:	Redacción del documento legal que formaliza los acuerdos alcanzados	- Personal Administrativo

| **Realised User Task**                     | **Elaboración de Contrato – Requerimiento: NGC-007**                                                                                                         |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Solicita la elaboración formal del contrato tras acuerdo en términos |
| **Participating Actors**       | Personal Administrativo |
| **Flow of Events**             | 1. El Ejecutivo selecciona una negociación con términos acordados<br>2. El Ejecutivo elige "Solicitar elaboración de contrato"<br>3. El sistema muestra resumen de términos acordados<br>4. El Ejecutivo verifica la información y añade notas específicas si es necesario<br>5. El Ejecutivo confirma la solicitud<br>6. El sistema registra la solicitud y cambia el estado de la negociación<br>7. El sistema notifica al Personal Administrativo sobre la nueva solicitud<br>8. El sistema genera un borrador preliminar basado en la plantilla y términos |
| **Exceptions**                 | - Si faltan términos obligatorios, el sistema impide continuar<br>- Si hay inconsistencias en los términos, muestra advertencias |
| **Preconditions**              | - La negociación tiene todos los términos principales acordados<br>- El Ejecutivo tiene autorización para solicitar contratos |
| **Postcondition**              | - Se crea una solicitud de contrato en el sistema<br>- La negociación cambia a estado "En elaboración de contrato"<br>- Se notifica al Personal Administrativo |
| **Includes Use Case**          | - Generación automática de borrador de contrato |
| **Used Services**              |  - Servicio de gestión documental<br>- Servicio de notificaciones<br>- Servicio de generación de documentos |
| **Non-functional requirements**| - El borrador preliminar debe generarse en menos de 5 segundos<br>- Debe guardar historial de todas las versiones del contrato |

### 11. Firma de Contrato

|-------------------------------|---------------------------------------------------------------------------------------------------------------------|
| **Initiating Actors**          | Ejecutivo Comercial: Coordinar y registrar la firma del contrato por las partes involucradas |
| **Participating Actors**       | CEO, Cliente (indirecto) |

| **Includes Use Case**          | - Carga de documentos firmados |
| **Used Services**              | - Servicio de gestión de citas<br>- Servicio de firma digital (opcional)<br>- Servicio de notificaciones |
